

# UI / PM Design Brief

## （由《个人健康数据系统 3.0 Architect 文档》派生）

> 本文档用于指导 **项目经理（PM）与 UI 设计**工作
> 不涉及算法实现、不涉及医疗判断
> 目标是：**让 PM 清楚地知道要设计哪些页面、每个页面的职责是什么**

---

## 1. 这个产品是做什么的？

这是一个**帮助用户长期查看和理解自己健康数据变化趋势的产品**，
重点在于：

* 看变化
* 看趋势
* 看风险提示

**不是**给出诊断结论，也**不**替代医生。

---

## 2. PM 可以假设的前提（非常重要）

在做 UI / 产品设计时，你可以默认：

* 所有健康数据**已经存在**
* 所有分析结果**由系统自动生成**
* UI **不需要**让用户理解医学细节
* UI **不需要**让用户编辑复杂数据

👉 你的工作是**展示与引导理解**，不是计算。

---

## 3. 产品中，用户“能看到”的核心内容是什么？

你可以把整个产品理解成只展示 5 类东西：

1. **健康指标**

   * 比如血压、血糖、体重等
2. **时间变化**

   * 每个指标都有“随时间变化的趋势”
3. **简单结论性描述**

   * 比如“最近三个月波动较大”
4. **风险提示**

   * 比如颜色、标签、提醒语
5. **健康报告**

   * 把一段时间的数据整理成一份总结

---

## 4. 建议的核心页面结构

> 以下是**推荐页面结构**，不是最终 UI 约束
> PM 可以在此基础上调整交互形式

---

### 页面 1：健康总览页（首页）

**这个页面解决的问题：**
👉 用户一打开，就知道「最近整体情况如何」

**页面应包含：**

* 关键健康指标卡片（如血压、血糖等）
* 每个指标的简要状态（稳定 / 波动）
* 简单风险提示（颜色或标签）

**不需要做的：**

* 不需要展示详细医学解释
* 不需要展示所有历史数据

---

### 页面 2：单一指标详情页

**这个页面解决的问题：**
👉 用户想看某一个指标“长期变化情况”

**页面应包含：**

* 指标名称与当前值
* 一张趋势图（按时间）
* 简短文字说明（如“近期有上升趋势”）
* 历史记录列表（按时间）

**不需要做的：**

* 不需要判断好坏对错
* 不需要给出医疗建议

---

### 页面 3：多指标分析页（可选，后期）

**这个页面解决的问题：**
👉 用户想知道“几个指标之间有没有一起变化”

**页面应包含：**

* 两个或多个指标的对比展示
* 简单关联说明（如“变化趋势相似”）

**注意：**

* 不需要解释“因果关系”
* 不需要复杂交互

---

### 页面 4：健康报告页

**这个页面解决的问题：**
👉 用户或医生想快速了解一段时间的健康情况

**页面应包含：**

* 时间范围（如最近 3 个月）
* 主要指标变化总结
* 风险提示说明
* 支持导出 / 分享

---

## 5. UI 文案与展示原则（来自架构的硬约束）

在所有页面中，请遵循以下原则：

* 使用“趋势”“变化”“可能”这类词
* 避免使用“诊断”“异常”“疾病”等词
* 所有数据必须带时间背景
* 不出现单一数值的绝对判断

👉 这是产品与医疗系统的**边界线**。

---

## 6. PM 不需要关心的事情（明确排除）

以下内容**不在 PM / UI 设计范围内**：

* 数据如何计算
* 算法是否准确
* 医学模型是否完整
* 指标背后的医学意义

这些由系统和后续专业人员负责。

---

## 7. PM 当前阶段的交付物

在当前阶段，PM 的目标是产出：

* 页面结构草图 / 原型
* 核心页面流转关系
* UI 设计说明

**不需要：**

* PRD 中的算法说明
* 医学专业论证

---

## 8. 一句话总结给 PM

> 你设计的不是“医疗系统”，
> 而是一个**让人看懂自己健康数据变化的工具**。
